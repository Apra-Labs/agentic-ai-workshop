Six Guardrails, Strongest First — Most People Build Backwards 5 min

Most people obsess over Level 6 while ignoring Levels 1–3. Build top-down.

Build From the Top Down

L1: ARCHITECTURE — Secrets in CI/CD. Branch protection. Limited git.
██████████ Structural — can’t be bypassed
L2: SANDBOX — Filesystem + network isolation (bubblewrap / seatbelt)
█████████░ OS-enforced — very hard to bypass
L3: HOOKS — PreToolUse blocks commands; PostToolUse validates
████████░░ Programmable + deterministic
L4: SETTINGS / DENY RULES — Disallowed tools, blocked commands
██████░░░░ Tool-enforced — requires config
L5: PROJECT INSTRUCTIONS — CLAUDE.md guidance, conventions
████░░░░░░ Agent-followed — advisory only
L6: PERMISSION PROMPTS — Interactive approval per operation
██░░░░░░░░ Human-dependent — approval fatigue is real

The Inversion: If L1 + L2 are solid, you can safely disable L6 (prompts). Hooks (L3) add automation. If architecture is weak, no amount of prompts will save you.

Hooks (Level 3) — Automation, Not Security

Unlike CLAUDE.md (advisory) or settings.json (static), hooks are programmable. But be honest: they catch mistakes, not attacks.

PreToolUse: Tiered response — hard block (exit 2) for irreversible, ask user (exit 0 + JSON) for risky, allow silently for safe.

PostToolUse: Auto-lint, auto-test — this is where hooks genuinely compound.

Configure in .claude/settings.json — matcher selects which tools trigger the hook. (Lab 4: full config + script walkthrough)

Three hook types: command (shell script), prompt (single-turn LLM check), agent (verification subagent).

Two Layers Make FSD Work

Layer 1: Permission Mode (system-level)
// .claude/settings.json
"permissions": { "defaultMode": "bypassPermissions" }

This is the real switch. Without it, the agent stops to ask “may I?” on every tool call. No amount of CLAUDE.md text overrides this.

Layer 2: Behavioral Instructions (CLAUDE.md)
## Working Style
Work autonomously. Don’t ask “should I proceed?” — just proceed. Explain what you’re doing as you go.
Commit after each logical unit of work.

This reduces conversational check-ins — the agent being polite. It’s a nudge, not a guarantee. The agent still pauses when genuinely uncertain.

Your role: Watch the terminal. Ctrl+C is your brake pedal. Inject ideas mid-stream: “try this approach instead.”

When Do You Actually Look?

FSD only works if you have natural observation points. They’re already in your workflow:

Moment What you do
Build / test running Glance at terminal — is it on track?
Commit message appears Read the diff summary — does it match intent?
Agent asks a question Answer or redirect — this is your steering input
Task boundary Review output before giving next task

Build wait times are natural observation windows — feature, not bug.

Ralph Wiggum Loop: Give It a Goal, a Promise, and a Limit 5 min

“I’m Helping!” — Your agent in an unbounded loop: enthusiastic but unaccountable.

The Ralph Wiggum Loop adds three constraints that turn enthusiastic flailing into bounded autonomy:

  • A clear finish line
  • Externalized state
  • A kill switch

The Plugin (Official Anthropic)

# Install once
claude plugin install ralph-loop@claude-plugins-official

# Usage pattern
/ralph-loop:ralph-loop "<goal + test + state + promise>" \
  --completion-promise "DONE" \
  --max-iterations 30

Full prompt example in Lab 5.

How It Works

Uses a Stop hook — when Claude tries to exit, the hook intercepts and feeds the same prompt back. Claude sees its own previous work (modified files, git history) and continues. Repeats until the completion promise is output or max iterations is hit.

Five Elements That Make It Work

State externalization“Update progress.md after each task”Progress lost on overflow

  • Creates init.sh (env setup)

↓
CODING AGENT (loops across windows)
while features remain:

  2. Pick ONE feature
  3. Implement + test
  4. Update feature_list.json
  5. Commit + update progress.md

Outer shell loop restarts on exit:

plan.md → feature_list.json

plan.md (human writes: phases, rationale) → feature_list.json (agent updates: "passes": true/false)

Why JSON? Markdown checkboxes get corrupted across context windows. JSON is unambiguous. (Lab 6: side-by-side walkthrough)

Key Design Decisions (Anthropic)

DecisionWhy
plan.md (human-readable)Approach, phases, rationale — the plan you already write
feature_list.json (machine)Derived from plan. Agent updates passes. JSON can’t be corrupted.
One feature per sessionHard stop after one feature. Prevents context exhaustion.
Git commits as checkpointsAgents can revert failed changes and recover
Shell loop + --max-turnsAuto restart. Requires Gold-tier safety (slide 10).

From One Agent to Sixteen — Three Levels of Parallelism 10 min

The key question at each level: who coordinates the work?

Level What it looks like Who coordinates? Status
1. Subagents One terminal, one conversation. You ask a question — Claude silently spawns helpers that research in parallel, then gives you one combined answer. The agent (internally) Stable, production
2. Agent Teams Multiple Claude sessions on your machine. A “lead” agent spawns “teammates,” each with their own context window. They share a task list and can message each other. A lead agent (delegates to peers) Experimental — research preview since Feb 5, 2026. Known bugs.
3. Independent Separate terminals, git worktrees, or even separate machines. Each agent works on its own branch. No built-in coordination at all. You (merge manually) Stable — just git

Honest take on Level 2: Agent Teams shipped Feb 2026 as Anthropic’s official answer to community tools like Tmux-Orchestrator. It works, but Delegate mode is currently broken (#25037) and it’s expensive. Most teams will get the best ROI from Level 1 + Level 3.

Our recommendation: Start with subagents (Level 1). When you need isolation, use git worktrees (Level 3). Watch Agent Teams mature — it’s the future, but not ready for production yet.

Deliberately Minimal Orchestration

while true; do
  claude --dangerously-skip-permissions \
    -p "$(cat AGENT_PROMPT.md)"
done

Agents self-coordinated through file-based locking and git as mutual exclusion.

Key Lessons

  1. Invest in test infrastructure, not orchestration — tests must be “nearly perfect”
  2. Design for agents, not humans — extensive READMEs, progress files, grep-friendly logs
  3. Minimize test output in context — important info goes to files, not stdout
  4. Handle time blindness — agents “will happily spend hours running tests”
